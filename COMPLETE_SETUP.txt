🎬 COMPLETE USER JOURNEY

Day 1 - Initial Login:
  1. User registers in owasp_app (standard registration)
  2. User logs in
  3. Backend queries security_lab.user_vulnerabilities
  4. No incomplete vulns found
  5. SELECT 8 random from vulnerabilities (ORDER BY RAND() LIMIT 8)
  6. Gets: [4, 9, 11, 15, 18, 20, 3, 7]
  7. Inserted into user_vulnerabilities with assignment_session=1
  8. Frontend stores in localStorage
  9. User can ONLY exploit those 8
  10. Others show 403 Forbidden

Day 1 - Exploitation:
  1. User tries IDOR_BALANCE (4) → IN THEIR LIST → Vulnerable ✓
  2. User tries PATH_TRAVERSAL (9) → IN THEIR LIST → Vulnerable ✓
  3. User tries MISSING_AUTH (16) → NOT in list → Safe (403) ✓
  4. System marks complete vulns in is_completed=1

Day 2 - Resume:
  1. User closes browser (data persisted in DB)
  2. User logs in again
  3. Backend checks user_vulnerabilities WHERE is_completed=0
  4. Finds SAME [4, 9, 11, 15, 18, 20, 3, 7]
  5. Returns same list
  6. User continues where they left off

Day 3 - Complete All:
  1. User completes all 8
  2. All marked is_completed=1
  3. User logs in again
  4. Backend checks: no incomplete vulns
  5. Creates new session (assignment_session=2)
  6. Selects 8 NEW different: [1, 5, 6, 10, 12, 14, 17, 19]
  7. Fresh challenge!

📊 DATABASE SCHEMA

vulnerabilities table:
  id (1-20)
  name (SQLI, IDOR_BALANCE, etc.)
  endpoint, description, difficulty

user_vulnerabilities table:
  id
  user_id (from security_lab.users)
  vuln_id (1-20, links to vulnerabilities)
  is_completed (0 or 1)
  completed_at (timestamp)
  assignment_session (1, 2, 3... for replays)

user_sessions table:
  id
  user_id
  session_number
  assigned_at, completed_at
  is_completed
